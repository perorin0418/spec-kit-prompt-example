# Spec Kit Prompt Example

**Spec Kit** は、AI を活用した仕様駆動開発を支援するプロンプトテンプレート集です。  
プロジェクトの「憲法(Constitution)」から「実装(Implement)」まで、段階的に設計・開発を進めることができます。

---

## 📚 概要

Spec Kit は以下の **7つのフェーズ** で構成されています:

```
1. Constitution (憲法)
        ↓
2. Specify (仕様)
        ↓
3. Clarify (要件明確化)
        ↓
4. Plan (計画)
        ↓
5. Checklist (チェックリスト)
        ↓
6. Tasks (タスク)
        ↓
7. Analyze (分析)
        ↓
8. Implement (実装)
```

各フェーズには **テンプレート** と **サンプル** が用意されており、AI にプロンプトとして渡すことで、段階的にプロジェクト設計を進められます。

---

## 🗂️ ファイル構成

| ファイル名 | 説明 |
|-----------|------|
| [`1.constitution`](1.constitution) | プロジェクト憲法のテンプレート |
| [`2.specify`](2.specify) | 仕様書のテンプレート |
| `3.clarify` | 要件明確化フェーズ |
| [`4.plan`](4.plan) | 実装計画のテンプレート |
| `5.checklist` | チェックリストフェーズ |
| `6.tasks` | タスク分解フェーズ |
| `7.analyze` | 分析フェーズ |
| `8.implement` | 実装フェーズ |

## 🚀 使い方

### ステップ1: 憲法(Constitution)を作成する

プロジェクトの **非交渉的な原則** を定義します。

1. [`1.constitution`](1.constitution) を開く
2. テンプレートの `＿＿＿＿` 部分を埋めていく
3. 参考として [`example/PhotoStream/1.constitution`](example/PhotoStream/1.constitution) を確認するとよいでしょう
4. AI に以下のように指示する:

```
/constitution
【ここに憲法(Constitution)を貼り付ける】
```

**ポイント**:
- 技術スタック、コーディング規約、テスト方針など、プロジェクト全体で守るべきルールを明記する
- この憲法は以降のすべてのフェーズで参照されます

---

### ステップ2: 仕様(Specify)を定義する

**何を作るか(WHAT)** を明確にするフェーズです。

1. [`2.specify`](2.specify) を開く
2. ペルソナ、ユーザージャーニー、受け入れ条件などを記述する
3. 参考: [`example/PhotoStream/2.specify`](example/PhotoStream/2.specify)
4. AI に以下のように指示する:

```
/specify
【ここに仕様(Specify)を貼り付ける】
```

**ポイント**:
- **技術的な実装方法は書かない**
- ユーザー体験と目的に焦点を当てる
- 測定可能な成功基準(KPI)を明記する

---

### ステップ3: 要件明確化(Clarify)を行う

仕様書で定義した内容のうち、曖昧・未決定・前提依存の部分を洗い出し、AIとの対話で明確化します。

1. AI に以下のように指示する:

```
/clarify
```

2. AI からの返答をもとに、要件を明確化する
3. 解決後、仕様書へ必要な反映 (差分のみ) を行う

**ポイント**:
- 曖昧さ: 用語定義不足 / ユースケース境界 / エッジケース / データ制約
- リスク: 期間超過要因 / 技術不確実性 / 外部依存 / セキュリティ懸念
- 未決定項目は Plan に流し込む前に最小限でも決着させる

### ステップ4: 実装計画(Plan)を立てる

**どのように作るか(HOW)** を設計するフェーズです。

1. [`4.plan`](4.plan) を開く
2. アーキテクチャ、ディレクトリ構成、非機能要件などを記述する
3. 参考: [`example/PhotoStream/4.plan`](example/PhotoStream/4.plan)
4. AI に以下のように指示する:

```
/plan
【ここに実装計画(Plan)を貼り付ける】
```

**ポイント**:
- 技術選定の理由を明記する
- テスト戦略、CI/CD、デプロイ計画も含める
- トレードオフと優先度を明確にする

---

### ステップ5: チェックリスト(Checklist)を作成する
実装計画をもとに、品質保証やリスク管理のためのチェックリストを作成するフェーズです。
1. AI に以下のように指示する:
```
/checklist
```

**ポイント**:
- 各チェック項目は具体的で測定可能にする
- テストケース、コードレビュー項目、セキュリティチェックなどを含める


---

### ステップ6: タスク分解(Tasks)

実装計画を具体的なタスク(T001, T002...)に分解するフェーズです。

1. AI に以下のように指示する:
```
/tasks
```

### ステップ7: 分析(Analyze)を行う

開発途中／完了後に成果物や進捗を振り返り、改善余地・学習ポイント・品質指標を抽出します。継続的改善サイクルを回す補助フェーズです。

1. AI に以下のように指示する:

```
/analyze
```

2. 必要に応じて憲法/仕様/計画へのフィードバック (破られた原則や新規原則案) を検討
3. 改善タスクは通常の Tasks リストへ取り込み再計画

**ポイント**:
- 分析は反省会ではなく「学習資産化」が目的
- 定量 + 定性 (例: バグ密度, リードタイム, NPS 風フィードバック)
- 改善案はコスト/期待効果/期間をセットで簡潔に

### ステップ8: 実装(Implement)を進める

実装を開始します。タスクに基づいてコードを書き、テストを実行し、レビューを行います。

1. AI に以下のように指示する:

```
/implement
```

2. タスクを順次実行し、進捗を管理する

**ポイント**:
- 定期的に進捗をレビューし、必要に応じて計画を調整する

## 📖 各フェーズの詳細

| フェーズ | 目的 | 成果物 |
|---------|------|--------|
| **1. Constitution** | プロジェクトの原則を定義 | 憲法ドキュメント |
| **2. Specify** | 何を作るか(WHAT)を明確化 | 仕様書 |
| **3. Clarify** | 要件を明確化 | |
| **4. Plan** | どう作るか(HOW)を設計 | 実装計画書 |
| **5. Checklist** | 品質保証・リスク管理のチェックリスト作成 | チェックリスト |
| **6. Tasks** | タスク分解 | タスクリスト |
| **7. Analyze** | 分析 | |
| **8. Implement** | 実装 | |

---

## ⚠️ 注意事項

- 各フェーズは **順番に進める** こと
- 憲法は **すべてのフェーズで参照される** ため、最初にしっかり定義しましょう
- 仕様書では技術的詳細を書かず、**ユーザー視点** を保つこと
- 実装計画で初めて **技術スタックや設計** を明記する
